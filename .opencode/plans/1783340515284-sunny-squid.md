# Plan: Godot MCP Setup for Eggbert

## Overview

Set up MCP (Model Context Protocol) capabilities for the Eggbert Godot project across all three tools: OpenCode, Zed, and Neovim. This involves configuring the `godot-mcp` server, enabling Godot's built-in LSP/DAP for GDScript, and ensuring C# (OmniSharp/Roslyn) LSP works properly with Godot's .NET SDK.

---

## Goal 1: Project-Level OpenCode Config + godot-mcp MCP Server

**What**: Create `.opencode/opencode.json` in the eggbert project root with godot-mcp as a local MCP server and a project-specific agent.

**Files to create**:
- `eggbert/.opencode/opencode.json` — project-level config
- `eggbert/.opencode/prompts/build.md` — Godot-aware build prompt (optional)

**Key decisions**:
- MCP server type: `local` (stdio), command: `npx @coding-solo/godot-mcp`
- `GODOT_PATH` env: `/usr/lib/godot-mono/godot.linuxbsd.editor.x86_64.mono`

**Config structure**:

```jsonc
{
  "$schema": "https://opencode.ai/config.json",
  "mcp": {
    "godot": {
      "type": "local",
      "command": ["npx", "@coding-solo/godot-mcp"],
      "enabled": true,
      "env": {
        "GODOT_PATH": "/usr/lib/godot-mono/godot.linuxbsd.editor.x86_64.mono"
      }
    }
  },
  "agent": {
    "build": {
      "instructions": [
        "This is a Godot 4.7 C# project (Eggbert).",
        "The godot-mcp MCP server is available for Godot operations (run project, launch editor, create scenes, etc.).",
        "Use `dotnet build` in the project root to compile C# scripts.",
        "C# files are in Godot's .NET SDK format (Eggbert.csproj uses Godot.NET.Sdk/4.7.0)."
      ]
    }
  }
}
```

**Verification**:
1. After creation, run `opencode` from the eggbert project dir
2. OpenCode should show godot-mcp as an available MCP server
3. Use a tool like `get_godot_version` via the agent

## Goal 2: Project-Level Zed Config + godot-mcp MCP Server

**What**: Create `.zed/settings.json` in the eggbert project root with godot-mcp as a context_server.

**Key context**:
- Zed uses `context_servers` (NOT `mcpServers`) in settings.json
- Local MCP servers use `command` (string) + `args` (array)
- Project-level `.zed/settings.json` overrides user-level settings

**Files to create**:
- `eggbert/.zed/` — directory
- `eggbert/.zed/settings.json` — project-level Zed config

**Config structure**:

```jsonc
{
  "context_servers": {
    "godot": {
      "command": "npx",
      "args": ["-y", "@coding-solo/godot-mcp"],
      "env": {
        "GODOT_PATH": "/usr/lib/godot-mono/godot.linuxbsd.editor.x86_64.mono"
      }
    }
  }
}
```

**Verification**:
1. Open eggbert in Zed
2. Open Agent Panel → Settings view
3. Confirm godot context server shows green indicator
4. Open a new agent conversation — godot tools should be available

## Goal 3: Neovim Godot-Specific LSP Enhancements

**What**: Add GDScript LSP (connecting to Godot's built-in LSP on port 6005), GDScript Treesitter grammar, GDScript DAP (port 6006), and verify C# LSP (OmniSharp) works with Godot's .NET project structure.

### 3a: GDScript LSP

**Files to modify/create**:
- `~/.config/nvim/lsp/gdscript.lua` — GDScript LSP config (connect to Godot LSP on port 6005)

GDScript LSP comes from Godot itself — start Godot, it opens port 6005. Neovim connects via TCP.

```lua
-- GDScript LSP — connects to Godot's built-in LSP server (port 6005)
-- Godot must be running with LSP enabled (default: port 6005)

local util = require('lspconfig.util')

return {
  cmd = vim.lsp.rpc.connect('127.0.0.1', 6005),
  filetypes = { 'gdscript', 'gd' },
  root_markers = { 'project.godot' },
  root_dir = util.root_pattern('project.godot', '.git'),
  -- ponytail: TCP connect only, no cmd needed — Godot runs the LSP server
}
```

**Register in Mason** or skip Mason registration (since Godot LSP isn't a Mason package — it's a built-in Godot service). Add to the lspconfig setup via `vim.lsp.enable()`.

### 3b: GDScript DAP

**Files to modify**:
- `~/.config/nvim/lua/plugins/dap.lua` — add Godot DAP adapter and GDScript config

Godot's DAP runs on port 6006 (default, configurable in Editor Settings > Network > Debug Adapter).

```lua
-- Add to existing dap config:
dap.adapters.godot = {
  type = 'server',
  host = '127.0.0.1',
  port = 6006,
}

dap.configurations.gdscript = {
  {
    type = 'godot',
    request = 'launch',
    name = 'Launch Scene',
    project = '${workspaceFolder}',
    launch_scene = true,
  },
}
```

### 3c: GDScript Treesitter

**Files to modify**:
- `~/.config/nvim/lua/plugins/treesitter.lua` — add gdscript, gdshader, godot-resource grammars

```lua
-- In treesitter config ensure_installed list, add:
'gdscript',
'gdshader',
'godot-resource',
```

### 3d: Verify C# LSP (OmniSharp) with Godot

**Check**: The existing OmniSharp config already works with Godot projects because:
- Godot's .NET SDK generates standard `.csproj` and `.sln` files
- OmniSharp reads `.csproj` to resolve GodotSharp NuGet packages
- `.NET 10` SDK is installed and `dotnet restore` works

**Action**: 
1. Run `dotnet build` in eggbert once to restore NuGet packages
2. Open any `.cs` file in Neovim — verify OmniSharp starts and provides completion
3. If OmniSharp struggles with `Godot.NET.Sdk`, may need to switch to `roslyn.nvim` (which works better with Godot SDK projects)

**Fallback**: If OmniSharp doesn't work well, add `roslyn.nvim` with a Mason registry extension:

```lua
-- In mason.lua, add registry:
registries = {
  'github:mason-org/mason-registry',
  'github:Crashdummyy/mason-registry', -- has roslyn for Godot
}

-- Then :Mason install roslyn
```

### 3e: Optional — External Editor Integration (Godot → Neovim)

Configure Godot's editor settings (manual, via Godot UI):
- Editor Settings > Text Editor > External > Exec Path: `/opt/nvim-linux-x86_64/bin/nvim`
- Editor Settings > Text Editor > External > Exec Flags: `--server /tmp/godot.pipe --remote-send ":n {file}:call cursor({line},{col}) "`
- Editor Settings > Text Editor > Behavior > Auto reload scripts on external change: `On`

Or simpler (no pipe): `{file}:{line}` — Neovim will open at the right position.

**Verification for Goal 3**:
1. Start Godot with eggbert project
2. Open a `.gd` file in Neovim — GDScript LSP should connect, showing completions/diagnostics
3. `:lua require('godot-lsp').status()` to verify connection
4. Open a `.cs` file — OmniSharp should start and resolve Godot types
5. Set a breakpoint and debug: `:lua require('dap').continue()` with Godot running

**Action**:

2. Open any `.cs` file — verify code completion, diagnostics, go-to-definition work
3. If Roslyn doesn't start automatically, add explicit config to `.zed/settings.json`:

```jsonc
  },
  "lsp": {

      "binary": {
        "path": "/path/to/OmniSharp",
        "arguments": ["-lsp"]
      }
    }
  }
}
```

**Verification**:
1. Open eggbert in Zed
2. Open `autoload/player/Player.cs`
3. Verify syntax highlighting, hover info, go-to-definition work

## Risks & Considerations

| Risk | Mitigation |
|------|------------|
| `godot-mcp` npm package may not be in npm registry (only published to GitHub) | Check if `npx @coding-solo/godot-mcp` works; if not, clone and build from source |
| Zed `context_servers` command format may differ between versions | Use the Agent Panel UI's "Add Custom Server" modal to generate correct config |
| `dpowling/godot-lsp.nvim` may conflict with existing `lspconfig` setup | It's standalone — uses `vim.lsp.rpc.connect` directly, no lspconfig dependency |
| OmniSharp struggles with `Godot.NET.Sdk` project format | Fall back to `roslyn.nvim` + Mason's community registry |
| Godot's built-in LSP requires Godot to be running | Document this requirement — LSP only works when Godot editor is open |
| OpenCode project config merges with global config | Project-level `opencode.json` merges with `~/.config/opencode/opencode.json` — only specify what differs |

## Post-Implementation: Manual Godot Editor Settings

After all automated setup is complete, open Godot Editor and configure:

1. **External Editor** (Editor Settings > Text Editor > External):
   - Exec Path: `/opt/nvim-linux-x86_64/bin/nvim`
   - Exec Flags: `{file}:{line}` (or leave empty for auto-detect in Godot 4.5+)
   - Check "Use External Editor"

2. **Auto-reload** (Editor Settings > Text Editor > Behavior):
   - Auto reload scripts on external change → `On`
   - Auto reload and parse scripts on save → `On`

3. **LSP/DAP ports** (Editor Settings > Network):
   - Language Server > Remote Port: `6005` (default)
   - Debug Adapter > Remote Port: `6006` (default)
   - Remote Host: `127.0.0.1`

4. **External editor for C#** (Editor Settings > Dotnet > Editor):
   - Editor: Select your preferred option (VS Code, etc.)  
   *Note: C# LSP is handled by OmniSharp/Roslyn in your editor, not Godot's built-in LSP*
